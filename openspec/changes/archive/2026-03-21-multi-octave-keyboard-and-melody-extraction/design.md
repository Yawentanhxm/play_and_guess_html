## Context

游戏当前音符数据只有 `note: 1-7`，键盘也只有一排 7 键，对应一个八度。大多数流行曲旋律跨越 1.5~2 个八度，导致很多音符被截断或映射错误。此外，MIDI 文件常含多轨道（主旋律 + 和弦伴奏），当前代码取第一个事件数≥10 的轨道，这未必是旋律轨。

## Goals / Non-Goals

**Goals:**
- 音符数据结构加入 `octave` 字段（-1/0/1），`note` 仍为 1-7
- 键盘扩展为三排 21 键，键位映射覆盖三个八度
- 简谱数字用 CSS 上下点标记八度
- 旋律轨道识别：从所有轨道中选出"最像主旋律"的一条

**Non-Goals:**
- 超出三个八度（更低或更高）的音域
- 自动转调到合适音域（超出范围的音符截断到最近八度即可）
- 多手弹奏（双手同时弹不同八度）

## Decisions

### 决策 1：音符数据结构 `{ note, octave, duration, midiNote }`

`midiNote` 已经包含完整音高信息（0-127），从中推导：
```
octave = Math.floor(midiNote / 12) - 5  // MIDI 60=C4=中八度 → octave=0
note   = scale.indexOf(midiNote % 12) + 1
```
八度范围限制为 -1/0/1，超界的钳制（clamp）到边界值。

### 决策 2：键盘键位映射

```
高八度（octave=1）: Q W E R T Y U  → 1̇ 2̇ 3̇ 4̇ 5̇ 6̇ 7̇
中八度（octave=0）: A S D F G H J  → 1  2  3  4  5  6  7
低八度（octave=-1）:Z X C V B N M  → 1̣ 2̣ 3̣ 4̣ 5̣ 6̣ 7̣
```

三排从上到下排列（高→中→低），与钢琴键盘视觉习惯一致。

### 决策 3：旋律轨道识别算法

评分综合三个维度，取最高分的轨道：
1. **同时发音密度低**（同一时刻只有一个音符在响）：权重高，旋律以单音为主
2. **音符总数多**：旋律比伴奏通常有更多独立音符
3. **音域居中**（平均 MIDI 音高在 55-80）：旋律通常是中高音区

### 决策 4：简谱八度符号渲染

用 CSS `::before`/`::after` 伪元素在数字上下加小圆点，而非 Unicode 合字，兼容性更好：
```css
.note.octave-high::after  { content: '·'; position: absolute; top: -8px; }
.note.octave-low::after   { content: '·'; position: absolute; bottom: -8px; }
```

## Risks / Trade-offs

- **旋律识别不完美**：某些编曲旋律在低音轨，纯算法难以 100% 准确 → 可接受，比当前随机取轨道好
- **键盘 21 键在小屏幕上拥挤** → CSS 缩放处理，三排标注清晰即可
- **旧存档的 note 数据无 octave 字段** → 游戏无持久化存储，无需迁移
